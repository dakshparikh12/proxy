# External audit findings (Gemini + 2× GPT frontier, deduped by theme) — triage input

*Three independent cross-family audits of the full v0-spec. Deduped and clustered below; each finding tagged with source(s). Triage rule: validate EACH against the ACTUAL current spec text (docs may already resolve it), then propose the SIMPLEST fix — **favor reverting to a simpler design over adding machinery; simplicity is the goal**. Never reduce the end goal (grounded in-meeting answers in ~1-2s, real verified work, honest notes, any-repo, ~$1/hr aspiration). Verdicts: VALID+FIX (give the simplest fix) / ALREADY-RESOLVED (cite where) / NOT-NEEDED (why) / SIMPLIFY (revert to a simpler system).*

## THEME A — Topology & structure (the #1 cluster; strong 3-way consensus)
- A1. **Topology contradiction (BLOCKER).** Doc 00 §4 says "GCE per-meeting compute + E2B; fits a VM/long-lived process better than Cloud Run's request model" and rejects "serverless-for-meetings"; Doc 04 builds its whole durability story on Cloud Run instance recycle/re-claim. The docs disagree on where the harness runs, where SDK state lives, the failure model. [Gemini-B1, GPT2-B1, GPT3-B3]
- A2. **Graph should live in Postgres, not SQLite (SIMPLIFY, strong consensus).** Doc 01 stores the dep graph in per-repo SQLite on the tenant volume; but Cloud Run can't mount that disk, and code_intel's `_read_one`/graph reads happen from the harness process → the flagship direct-answer path "has no legal way to reach its own data." Fix: `graph_nodes`/`graph_edges` as two Alembic Postgres tables — any process queries them, the reachability problem evaporates, per-file `DELETE WHERE file_path=?` re-derive works identically. [GPT2-B1]
- A3. **Collapse 5 services → 3 deployables (SIMPLIFY, consensus).** transport/scribe/orchestrator/workroom share one meeting lifecycle, one event sequence, one budget, one mouth — deploying independently adds network seams + distributed failure with no scaling need. Proposed: **control_plane** (Cloud Run: auth, connect, webhooks, artifacts, approval, reconcile) · **meeting_runtime** (Cloud Run min≥1, no-cpu-throttle: transport+Scribe+Proxy+Workroom-host-shell as internal PACKAGES not services) · **code_intel** (GCE + encrypted persistent storage: clones/indexes/LSP, tenant+SHA-scoped internal API) · E2B external · Cloud SQL/GCS durable state. [GPT2-B1, GPT3-B3/S8]
- A4. **Direct-answer reads execute in the warm E2B sandbox** (pinned snapshot, one ~50-100ms hop), re-scoping CANONICAL §11.6's "no E2B round-trip" to mean "no Workroom *session*" — which is what the latency argument actually requires. [GPT2-B1]
- A5. **Pinned meetings don't have a pinned graph (BLOCKER).** Meetings pin a SHA but push handling updates graph rows in-place → a SHA-A meeting queries SHA-B-derived graph. "Code never shifts underneath it" is false for the graph. Fix: SHA-addressed immutable graph versions `(repo_id, commit_sha, graph_version)`; atomically publish a pointer; retain versions referenced by active meetings; GC unreferenced. [GPT3-B7]

## THEME B — Durability & state (the "no-bus but durable" gaps; GPT3 most rigorous)
- B1. **No durable event inbox (BLOCKER).** Runtime uses `asyncio.create_task` + in-process queues + completion callbacks; `operation_runs` records op lifecycle but is NOT an event inbox. A process dying after receiving a Recall webhook / control command / Workroom result but before Proxy consumes it → the event is lost on recovery. Fix: ONE Postgres `meeting_events` table (inbox/outbox: id, meeting_id, seq, event_type, payload, dedupe_key, state, claimed_by, claim_expires_at) — all external callbacks + task completions append first; owner drains in sequence; LISTEN/NOTIFY may wake but the table is source of truth. NOT an external bus. [GPT3-B4]
- B2. **Meeting ownership has no lease/fencing token (BLOCKER).** 30s heartbeat → 5min stale → 5min reconcile can block recovery for minutes; worse, a recovered old owner can still emit speech/side-effects (no fencing). `created_by` identifies but doesn't fence. Fix: `owner_instance_id`, `fencing_epoch` (atomic increment on claim), `lease_expires_at`; heartbeat 5-10s / lease 30-45s; every ack/delivery/mutation carries `(meeting_id, fencing_epoch)` and fails if not current. [GPT3-B5]
- B3. **Workroom crash recovery asserted but not specified (BLOCKER).** "re-picked by the drain loop / boot-recovery guarded by a completion predicate" — no drain interface, claim query, task payload schema, retry policy, completion-predicate format, or result-delivery event defined. Fix: canonicalize `operation_runs.progress` schema OR a `workroom_tasks` table with: immutable task bundle+hash, sandbox id + pinned SHA, state enum, claim lease + attempt, artifact/result refs, completion-predicate type+value, final-envelope event id, retryable-error class + next-attempt; define the exact claim/recovery SQL. [GPT3-B6]
- B4. **Notes deltas lack replay idempotency & revision control (BLOCKER).** Applier mints process-local ids (c14) + free-form patches by target id; no source-event id, expected-revision, dedupe window, or typed field patch. Webhook redelivery / repeated Scribe call / recovery → duplicate/reorder. Fix: every delta needs delta_id, source_event_id UNIQUE, meeting_revision, expected_entry_revision?, entry_id, typed op union; apply transactionally under the single-writer lock; reject/rebase stale revisions; typed patch models per entry kind. [GPT3-B15]
- B5. **transcript_segments.status DEFAULT 'comprehended' is backwards (SHOULD).** A crash between append and Scribe-process leaves rows falsely "comprehended" so the close backfill never repairs them. Fix: default `'pending'`; applier flips to 'comprehended' on delta apply; close backfills pending+gap. [GPT3-S10]
- B6. **Close path unbounded + can miss its only delivery channel (BLOCKER).** One strong-model pass over full raw transcript + ledger, then post to chat, then teardown — multi-hour meetings have no size threshold; meeting can end before the pass finishes or chat delivery succeeds; `if_generation_match=None` doesn't express create-only. Fix: persist `close_jobs`/idempotent close before teardown; chunked-extract→reduce over a token threshold; create with generation-match 0; post a durable provisional notes URL before leaving; finish enrichment async; authenticated post-meeting retrieval on chat-delivery failure. [GPT3-B16]
- B7. **Durable webhook queue referenced, never defined (SHOULD).** "webhook queue" (00 §4) + "enqueued durably, deduped by delivery-GUID+SHA" (01 §3.6) has no table. Fix: `webhook_events(id, provider, delivery_guid UNIQUE, sha, payload, status, received_at)`. [GPT3-S11]

## THEME C — Agent-loop & delivery seams (the runtime bugs a loop builds verbatim)
- C1. **stream_deltas applied TWICE → silent text loss (BLOCKER).** Doc 04 `BehaviorRunner.run()` yields `stream_deltas(raw)`; Doc 08 handler re-wraps `stream_deltas(ctx.orchestrator.answer(...))`. Double-application drops everything shorter than running max → voice/chat show only "Hello ". CANONICAL §11.3 pins the signature but not where it's applied once. Fix: one sentence in §11.3 "applied exactly once inside BehaviorRunner.run(); downstream MUST NOT re-wrap"; delete the Doc 08 §4.5 wrap. [GPT3-B2]
- C2. **Two incompatible delivery models (BLOCKER).** Doc 04 = agent-tool-driven (`speak`/`send_chat` tool calls). Doc 08 §4.5 = stream-projection-driven (`ctx.orchestrator.answer()` + projector auto-extracts first sentence to voice). Both → Proxy speaks twice; only-projector → no input; and `handle_meeting_turn`'s append_user/append_assistant writes chat-app turns into `transcript_segments` (conflates meeting transcript w/ conversation log — shaped for a chat product). Also: no doc specifies the CARRIER for the AgentChunk/progress stream harness→transport projector→tile WS. Fix: pin wake-turn tools as the ONLY delivery authority; narrow projector to TOOL_USE→tile lines / structured TOOL_RESULT→canvas / optional TTS-streaming of speak() deltas; carrier is in-process (transport = package in meeting_runtime); delete/rescope `handle_meeting_turn`. [GPT3-B3]
- C3. **answer-question.yaml sample can't direct-answer (BLOCKER).** The normative sample behavior declares `tools: [dispatch_workroom, speak, send_chat, show_screen, ack, cancel_task]` — NO code_intel/batch_read — so it can ONLY dispatch, contradicting §3.4 prose + CANONICAL §11.6. In a spec-driven build the example IS the spec. Fix: regenerate the sample with `mcp__code_intel__{get_dependents,who_writes,list_entry_points,batch_read,grep,read}` in config.tools + the code_intel mcp_servers mount; fix every "every ask → workroom" narrative. [Gemini-implied, GPT2-B2, GPT3-B5]
- C4. **Scribe raw-Messages-API vs SDK-adoption mandate (SHOULD/SIMPLIFY).** Doc 03 runs the Scribe as a raw `messages.create` w/ manual cache breakpoint; Doc 00 mandates the unmodified SDK. Two auditors: use SDK primitives structured to hit caching, OR keep raw but acknowledge it bypasses the global retry/telemetry wrappers. [Gemini-#5/#7, GPT2-#4] *(counter: raw call is deliberate for the 1-2s manual-cache micro-call; verify which is actually simpler+cheaper.)*
- C5. **Parallel workers concurrently resuming ONE shared SDK session (SHOULD).** Doc 05 describes fresh query() calls resuming the shared session → history ordering / context contamination / provider session races. Fix: one lead session + one independent child session per worktree given an immutable plan slice; structured result back to lead; no concurrent writes to one session. [GPT3-S4]
- C6. **Large-task Workroom path is over-engineered for a lean core (SIMPLIFY, consensus).** Core includes plan-critic + parallel worktree workers + separate verifier + up to 3 specialist verifiers + red-team panel + merge + agreement-confidence + bounded replan (doc says fan-out ~15× tokens). Fix: core = one lead session + persisted plan (multi-file) + SEQUENTIAL default + ONE fresh-context verifier + deterministic evidence gate; fan-out only for explicitly-independent changes above a measured duration threshold; move specialist panel + red-team to Expansion. [GPT3-S6]
- C7. **YAML behavior registry is over-indirection for a handful of behaviors (SIMPLIFY).** Inherited from a ~90-task-type repo; Proxy has a few wake behaviors; and it's the SOURCE of the direct-answer contradiction (C3). Static Python `CAPABILITIES` also has service-binding strings a TS frontend can't import. Fix: typed Python `BehaviorConfig` constants in V0 (judgment stays in-model — removing YAML doesn't hard-code the choice); generate a small JSON/TS capability manifest at build for UI labels. [GPT3-S7] *(revert-to-simpler candidate — weigh vs the "config not code" principle.)*
- C8. **The deterministic evidence gate is deterministic but not evidentiary (SHOULD).** It regexes model-authored text (`exit ... 0|pass(ed|ing)?`) — "the tests did not pass" contains "pass"; a model can claim "exit code 0". Fix: sandbox tool transport emits host-trusted structured receipts (command_id, argv, exit_code, stdout_ref, artifact_hashes); the gate reads receipts + parser results + file hashes, not prose. [GPT3-S3]

## THEME D — Code intelligence & graph correctness
- D1. **Referent matcher relies on the DEFERRED agentic map (BLOCKER).** Doc 03 binds named things via "Doc 01's map (area names, file names, key symbols)" — but the L1/L2 capability map is Expansion. Fix (simplest): define a minimal `lookup_referent(term)→{node_id|area|None}` over `graph_nodes` + the overview's area names, owned by services/code_intel; unmatched stays unbound; OR push referent resolution to the Workroom agent's native search at task time. [Gemini-#2/#4, GPT3-S4]
- D2. **who_writes / "any language, any scale" overclaims (BLOCKER).** Mechanical tree-sitter+LSP can't derive ORM table writes (Django/SQLAlchemy/Rails/Prisma/query-builders/stored-procs/string-SQL); CANONICAL itself flags this as a top risk. Fix: a V0 SUPPORT MATRIX — tier 1 exact-supported (named lang/framework w/ implemented extractors), tier 2 symbol-exact + data-flow-lower-bound (LSP refs work, table writes don't), tier 3 search-only; unsupported stacks NEVER emit an exact who_writes; replace "any language/scale" with tested limits + degradation. [Gemini-riskiest, GPT2-риск, GPT3-B9]
- D3. **Zoekt referenced but no infra (SIMPLIFY, consensus).** Doc 01 §3.6 builds a Zoekt trigram index >500k LOC; Doc 00 Terraform has no Zoekt cluster. Fix: **cut Zoekt from V0, use ripgrep on the clone** (accept slight big-monorepo latency). *(GPT2 counter: `zoekt-mcp` server as an MCP tool avoids custom orchestration — but that's still infra; simplest is ripgrep-only for V0.)* [Gemini-#3/#5, GPT2-#2]
- D4. **Changed-file-only graph invalidation is semantically wrong (BLOCKER).** Re-extracting only changed files' rows misses edges stored FROM unchanged callers (rename in A → B's stale edge remains). Fix (simplest+correct): **full graph rebuild on each relevant push for pilot-sized repos**; add incremental only after measurement; force full rebuild on force-push / parser upgrade / large change-set. [GPT3-B8]
- D5. **batch_read / get_dependents defaults are token bombs on the hot path (SHOULD).** 10 files × 500 lines ≈ 40-60K input tokens/answer (~$0.12-0.18 Sonnet) — one heavy answer exceeds the whole orchestration line; a transitive `_closure()` on a hot symbol returns thousands of nodes into the wake turn. Fix: `max_lines_per_file` default ~150; `get_dependents(limit≈50)` top-N by PageRank + truncation count. [GPT3-S8]
- D6. **Tool contract is not singular (BLOCKER).** Docs alternate: native grep/read · host-side code_intel MCP · sandbox MCP grep/read · LSP in the sandbox · LSP on persistent infra; tool lists differ; some include deferred tools, omit batch_read. Fix: ONE canonical matrix (tool | executes-where | mutable | required-args | dispositions); remove duplicate names + implicit routing. [GPT3-B10]
- D7. **Readiness too weak for the trust promise (SHOULD).** Low parse-coverage floor + file-count coverage weights generated files equally. Fix: require 100% files classified (supported/excluded/unsupported); 100% parse for exact-supported (ex generated/vendor); separate edge coverage; visible capability tiers; readiness tied to exact pinned SHA; `not_ready` when exact-supported critical files fail. Add `indexed_at`/`pinned_sha` to the readiness surface. [GPT3-S17, +nice-to-have]
- D8. **table-node id match is fragile.** `endswith("::refunds")` collides with a method named `refunds`. Fix: pin id format `table::<name>` and match `kind=="table"`. [GPT3-nice]

## THEME E — Cost & latency
- E1. **The $1/hr promise is arithmetically contradicted (BLOCKER).** transport $0.75-0.85 + Scribe $0.10-0.20 + orch $0.02-0.08 = $0.87-1.13 BEFORE Workroom/E2B/compute/storage/egress, yet rolled into "$0.95-1.15" while also doing 3-8 builds. Fix: one honest contract — **"listening baseline ~$1/hr; substantive tasks separately metered task budget"** (recommended) OR make all-in $1 non-negotiable and change transport economics first. Don't let the breaker disable defining functionality to preserve marketing arithmetic. [GPT3-B11, echoed]
- E2. **The meter doesn't cover what it enforces (BLOCKER).** `meeting_cost` = model spend mostly; no accounting for Recall time / STT / TTS chars / E2B runtime / compute / storage / egress; no pre-spend reservation (a big task crosses the hard cap before next check). Fix: `meeting_cost_entries(category, provider, quantity, unit, cost_usd, source_event_id)`; continuous time-accrual for transport/compute; actual char/audio for TTS/STT; reserve max cost before a task; settle reservation vs actual; reject/ask-approval when reservation > remaining budget. [GPT3-B12, Gemini-S1]
- E3. **Scribe cost figure fails its own arithmetic (SHOULD).** ~$0.0015/call × 200 + ~40 summary refreshes + 10% gate ≈ $0.30-0.35/hr vs claimed $0.10-0.20. Fix: restate envelope $0.15-0.35/hr or lengthen coalescer windows. [GPT3-S9]
- E4. **Latency SLA fork (BLOCKER/SHOULD).** 2.5s gate is impossible for LSP-bound answers (STT ~300ms + LSP ~2s + inference + TTS + Output Media). "Response" means first-text vs first-audio vs complete inconsistently. Fix: define the 2.5s gate strictly for SHALLOW grep/graph-only queries; LSP-bound answers use the tile "checking…" ACK, exempt from the 2.5s voice gate. Pin measurable SLOs: ack-audible p95≤500ms; first-grounded-text p50≤2s/p95≤4s; first-grounded-audio p50≤2.5s/p95≤5s; direct-path-one-gather-pass only; record tool/turn count per sample. [Gemini-#1, GPT3-S1]
- E5. **Smart Turn v3 likely redundant (SIMPLIFY, medium-confidence).** AssemblyAI Universal-Streaming ships end-of-turn detection on the stream you already pay for; keep Silero for <200ms barge-in, drop the 2nd turn model, re-add if boundary quality suffers. [GPT3-over-eng]

## THEME F — Security & access
- F1. **Authenticated approval/review surface is missing (BLOCKER, strong consensus).** The tile is a video stream (can't click); connect page is setup-only; draft card "approve = your click" + notes-link have nowhere to live; GCS signed URL dies ≤7d (kills "forward to the VP"); public URL leaks code. [Gemini-implied, GPT2-B4, GPT3-B13]
- F2. **Fleet-shared HS256 secret lives inside untrusted sandboxes (BLOCKER).** The verifier holds the shared signing secret and runs in E2B which executes arbitrary repo code → read `/proc/<pid>/environ` → mint tokens for any session_id → cross-tenant. Fix: per-sandbox random secret minted at provision (host keeps the map) OR asymmetric (host signs RS256/EdDSA, sandbox holds only the public key). Update S9 pass condition. [GPT3-B6-sec]
- F3. **Delete the clickable-tile inbound path (SIMPLIFY).** Humans can't click a video stream; `TILE_ADDRESS` inbound + tile-originated `ChannelAction` is dead/spoofable. Voice/chat already carry every control. Fix: tile is outbound-only; delete TileAddress; split inputs = Recall callbacks→durable events / authenticated review+connect→HTTP / tile→outbound signed state only. Tile page gets a meeting-scoped bearer token in the URL handed to Recall (cookie auth can't admit Recall's headless browser). [GPT2-B4, GPT3-S10]
- F4. **Core apply: PR vs notes-edit disagreement (BLOCKER).** CANONICAL §4 says core apply = notes-edit; world-touching applies (PR) = Expansion (read-only scope). Doc 05 repeatedly promises "staged PR"; Doc 09 S3 says accept "applies" it. [GPT3-B14]
- F5. **Sandbox egress default-deny vs functionally-open (SHOULD).** Sandbox promises default-deny egress while the toolbelt promises web research / Playwright / package ops / connectors. Fix: web fetch/search + connectors execute HOST-side returning bounded data; package install via allowlisted registry proxy or pre-baked deps; browser = separately-authorized tool w/ domain allowlist; no arbitrary E2B outbound in core. (Conforms with CANONICAL §10 SKIP of Tool-Search/code-exec-MCP.) [GPT3-S12, S7]
- F6. **propose_change can't represent a multi-file draft (SHOULD).** Signature single-target; row `kind='file-change'` singular; but S3/demo stage a multi-file PR-shaped draft; shown code reads a dead `review_session` on a host that can't read sandbox files. Fix: `propose_change(kind, summary, files:[{path,old_sha?,new_content}]|unified_diff)` → one GCS bundle per draft; original from the agent or the canonical clone at pinned SHA; delete the review_session remnant. [GPT3-S2]
- F7. **Tenant offboarding shreds the volume but not Postgres/GCS (SHOULD).** Crypto-shred covers clones/indexes; transcripts/note_deltas/notes/drafts live outside the envelope key. Fix: offboard sweep (delete tenant Postgres rows + GCS prefixes) in the reconcile family + a retention default (transcripts ~90d). [GPT3-S18]
- F8. **Consent / participant-authority / data-lifecycle undefined (BLOCKER→policy).** No behavior on objection / organizer-removes-consent / late-joiner-can't-see-chat; anyone can ask/patch-notes/issue-controls while only "a named human" approves. Fix: a core policy matrix (ask=any; private catch-up=attributed; cancel-own-task=asker; stop-all/end-bot=organizer/tenant-member; final-note-override=organizer audibly-ack; accept-draft=authenticated tenant role) + consent states + retention/deletion defaults + audit retention. [GPT3-B17]
- F9. **Private artifact delivery has no access contract (SHOULD).** Notes/drafts to GCS + chat link — auth/expiry/tenant enforcement undefined. Fix: authenticated artifact proxy OR short-lived signed URLs issued only after server-side tenant check; log access, expire, revoke; raw GCS URIs never user-facing. [GPT3-S18/N-provenance]
