# Final optimization pass — the last gaps before a full 00→09 run
### Reviewed the whole loop end-to-end. Five real optimizations remain; all are added below. After these, the trust machinery is complete for the free/simulated evidence tier.

## 1. "Done" is a fixed bar, not builder confidence — the Completeness Sweep (Phase 7.5)
**The correction to "keep iterating past green until confident":** open-ended subjective iteration re-collapses maker≠checker and risks over-engineering. Replaced with a bounded, independent mechanism — after all criteria are green, a **fresh-context agent re-reads the entire spec** (not the criteria) and lists any behavior no criterion covers. A gap → new criteria → re-seal → build. The doc is complete only when the sweep finds nothing uncovered. This *achieves your goal* ("nothing left to do, exactly the spec") through the criteria mechanism, so "done" stays trustworthy and terminates. Wired as ORCHESTRATION Phase 7.5.

## 2. Density tiers for serious docs (EXHAUSTIVE-COVERAGE amendment)
Cosmetic behaviors get one criterion; **serious behaviors** (security, concurrency, money, irreversible writes, routing, staged-draft gates, latency SLOs) get their **exact accept/reject boundary decomposed** — boundary values (at/below/above the limit), exact typed error/tag shapes, ordering & atomicity, one criterion per decision branch, and an *enumerated negative frontier* (the inputs that must be rejected, stated as precisely as the accepted set). A reader must be able to reconstruct the exact accept/reject line from the criteria alone. Docs 04 (orchestrator routing) and 05 (workroom task classes) are where this matters most.

## 3. Cross-doc regression gate (ORCHESTRATION Phase 8)
Later docs import earlier docs' code (04 uses 01/02/03; contracts in libs/contracts are shared). A previously-green doc can silently break when a later doc lands. **Before advancing, re-run rung-1 for docs 00..N-1; all must stay green.** Without this, a full 00→09 run can finish "green" with an earlier doc quietly broken. This is the one gap that specifically bites a *chained* build.

## 4. pass^k determinism gate (concurrency & nondeterminism)
Any criterion whose behavior is concurrent, ordered, or otherwise nondeterministic runs **k times (k≈5); all k must pass** before it counts green. A criterion that passes 4/5 is a flake = a real defect (races, ordering bugs), not luck. Cheap (no API — these are deterministic-code tests), and it's the only thing that catches the race conditions that single-run tests miss. Applies especially to the atomic-swap, webhook-idempotency, and multi-server criteria.

## 5. Freeze the authority per doc (spec + bundle immutability)
The spec a doc is built against must be **hashed and frozen** at Phase 1, alongside the sealed bundle. If the spec text changes mid-build, criteria and code silently reference different truths. The manifest already hashes the bundle; extend it to hash the exact `product/v0-spec/<doc>` + the CANONICAL sections it cites. A spec change = a new authority version = prior evidence void (same discipline as the bundle). Prevents the subtle "we edited the spec and now green means nothing" failure.

## The final definition of DONE for a doc (all must hold — none is the builder's word)
1. RTM coverage gate green (every requirement → criterion).
2. Every blocking criterion green under `verify.sh`, at **pass^k** where nondeterministic.
3. Mutation gate: no surviving mutants on blocking-criterion code.
4. Workflow simulations pass as real execution traces.
5. Fresh-context verifier returns `VERDICT: DONE` (adversarial, cross-checks commits vs diff).
6. **Completeness sweep** (fresh context, spec-vs-criteria) finds zero uncovered behavior.
7. Cross-doc regression: docs 00..N-1 still green.
8. Integrity + authority hashes unchanged (bundle and spec never weakened mid-build).

Only then does the doc advance. This is the "nothing left to optimize, exactly the spec" bar, expressed as fixed checks instead of a feeling — so it terminates, and so it can be trusted.
